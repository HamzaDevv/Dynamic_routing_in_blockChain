What is PCER-T?

PCER-T (Priority-Conscious Energy-Resilient with Trust) is the synthesised final protocol. It takes the three strongest capabilities from the research landscape and fuses them into a single cost function:

QoS Tag-Weights (Proposed PCER) Trust + NFR Blacklist (RPL-MRHOF) ETX Penalty (MRHOF / RSOF) Soft Energy Floor (PCER) Load Balancing (T-OF / MNP-OF)
Synthesised from: RSOF (Alotaibi et al. 2025) · BLET-IoT (J.Supercomputing 2025) · T-OF · SMTrust-RPL (Sensors 2022) · Proposed PCER

PCER-T Cost Function

For each candidate next-hop node v, cost is computed as:

// Hard guards (→ Infinity if violated)
IF nfr_blacklisted(v) → ∞
IF battery(v) < 3% → ∞

// Soft demotion zone
energy_penalty = battery(v) < 8%
                ? (0.08/battery(v)) × 25
                : (1/battery(v)) × 2

// Tag-adaptive weights
w_delay = [CRIT:0.85, STD:0.45, BULK:0.05]
w_energy = [CRIT:0.05, STD:0.30, BULK:0.65]
w_trust = [CRIT:0.08, STD:0.18, BULK:0.22]
w_lb = [CRIT:0.02, STD:0.07, BULK:0.08]

// Sub-costs
delay_cost = (edge_ms + h_dist) × w_delay
trust_cost = (1 − trust(v)) × 40 × w_trust
lb_cost = load_penalty(v) × w_lb

cost = delay_cost + energy_penalty×w_energy
       + trust_cost + lb_cost

Why Each Ingredient Matters

Tag-Weights
Proposed PCER used binary all-or-nothing weights. PCER-T uses fractional blends — Critical still strongly prefers speed but doesn't zero out energy, preventing it from burning a dying node.
Trust Guard
Trad. PCER and Proposed PCER had zero concept of node reliability history. PCER-T tracks forwarding PDR per node and blacklists chronic droppers, just like RSOF's NFR exclusion.
Soft Floor
Old PCER had a hard 5% cutoff — nodes fell off a cliff. PCER-T has a demotion zone at 8%: nodes get increasingly penalised before being hard-blocked at 3%, preventing sudden route flaps.
Load Balance
Inspired by T-OF's NOCS metric. When two hops are near equal cost, PCER-T prefers the one that has forwarded fewer packets recently — spreading wear across the mesh.
Hard guards from RSOF NFR exclusion | Soft floor from BLET-IoT energy-aware trust | LB from T-OF NOCS metric

PCER-T supersedes both previous PCER variants

What PCER-T Does NOT Have (Future Work)

To keep it implementable on constrained IoT hardware, PCER-T omits: cryptographic trust verification (blockchain), topology-level DIO message overhead management (full RPL), and TSCH slot scheduling. These remain open research directions.

Ref: RFC 6550 DIO overhead | 6TiSCH TSCH | RPL-BlockTrust (2025)
